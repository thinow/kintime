# DNS — thinow.dev

Domain registrar: **Porkbun** · DNS provider: **Cloudflare** (via Porkbun's default integration)

## Records

### Porkbun defaults (catch-all)

| Type  | Host         | Value                  |
|-------|--------------|------------------------|
| ALIAS | thinow.dev   | pixie.porkbun.com      |
| CNAME | *.thinow.dev | pixie.porkbun.com      |

These are Porkbun's default records. The `ALIAS` handles the apex domain; the wildcard `CNAME` catches any subdomain not explicitly defined. Neither serves Kintime — they just prevent unresolved DNS errors on unregistered subdomains.

### Kintime frontend (Vercel)

| Type  | Host                  | Value                                    |
|-------|-----------------------|------------------------------------------|
| CNAME | kintime.thinow.dev    | f58238abd2adcb84.vercel-dns-017.com.     |

Routes `kintime.thinow.dev` to Vercel. This value was generated by Vercel when the custom domain was added in the Vercel project settings. Vercel handles TLS automatically once this record resolves.

> The backend (`kintime-api.fly.io`) uses Fly.io's default subdomain — no custom DNS record needed for it.

### Email sending (Resend)

Resend sends magic link emails from `no-reply@mail.kintime.thinow.dev`. Three records authenticate the emails and prevent them landing in spam:

| Type | Host                                        | Value                                        | Purpose               |
|------|---------------------------------------------|----------------------------------------------|-----------------------|
| MX   | send.mail.kintime.thinow.dev                | feedback-smtp.eu-west-1.amazonses.com (p=10) | Resend uses Amazon SES (EU region) as its sending infrastructure |
| TXT  | resend._domainkey.mail.kintime.thinow.dev   | p=MIGfMA0G…                                  | DKIM — cryptographic signature on outbound emails, proves they weren't tampered with |
| TXT  | send.mail.kintime.thinow.dev                | v=spf1 include:amazonses.com ~all            | SPF — authorises Amazon SES to send on behalf of this domain |
| TXT  | _dmarc.thinow.dev                           | v=DMARC1; p=none;                            | DMARC — monitoring mode (`p=none`): collects reports but doesn't reject or quarantine failing emails |

> `p=none` is the safe starting policy. If deliverability becomes a concern, escalate to `p=quarantine` then `p=reject` once SPF/DKIM are confirmed working.

### SSL certificate (ACME challenge)

| Type | Host                       | Value          |
|------|----------------------------|----------------|
| TXT  | _acme-challenge.thinow.dev | owTVgJ0S…      |
| TXT  | _acme-challenge.thinow.dev | PuCvnXHE…      |

DNS-01 ACME challenge records used to prove domain ownership when issuing or renewing a TLS certificate. Two records can coexist during a renewal (old + new challenge). These are managed by whichever tool requested the certificate and can be deleted once the certificate is issued — though leaving them doesn't cause harm.
